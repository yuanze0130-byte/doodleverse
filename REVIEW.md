# REVIEW

这份 REVIEW 不是“总结成绩单”，而是基于当前版本做的真实反思，目的是帮助后续迭代少走弯路。

## 一、当前版本的判断

当前项目已经从“概念 Demo”进入了“可持续迭代的原型阶段”。

它已经具备了一条基本可走通的链路：

- 在白板里组织素材
- 在底部主输入框里输入 Prompt
- 用 `@` 引用白板元素
- 通过右侧导入参考图
- 进行 Prompt 润色 / 生图 / 生视频
- 把结果回流到白板或右侧历史区

这说明产品的主骨架已经形成。

但它还没有进入“稳定产品阶段”。目前的问题已经从“有没有功能”转移到了“功能之间是否统一、是否稳定、是否可维护”。

## 二、这次迭代做对了什么

### 1. 白板主工作流方向是对的

现在的结构已经比较接近正确的交互关系：

- 左侧负责工具和图层
- 中间负责画布
- 底部负责主 Prompt
- 右侧负责参考图、历史生成和素材辅助

这比之前“多个区域都像在抢主输入权”的状态要清晰很多。

### 2. 主输入框回到了产品中心

把底部输入框做成大面积、可伸缩、可展开功能项，是对的。  
用户的主要创作行为应该围绕它发生，而不是让右侧输入框承担主工作流。

### 3. API 配置开始和实际能力挂钩

以前设置面板里只是“存一个 key”，现在开始按 `LLM / 图片 / 视频 / Agent` 分类，至少在信息架构上更接近真实产品了。

### 4. 生成历史开始形成闭环

右侧历史生成自动保存到本地并支持拖回白板，这一步很关键。  
这让“生成结果不是一次性对话产物，而是白板资产”的方向更加明确。

## 三、这次迭代暴露出来的问题

### 1. UI 调整和功能接入被混在一起推进

这一轮里我们同时在做：

- UI 仿制
- 输入框重构
- `@` 机制恢复
- 设置面板重写
- 多 provider 路由接入

这些事情叠在一起，导致中间过程反复出现“界面像了，但功能断了”或者“功能接上了，但输入框坏了”的情况。

后面更稳的方式应该是：

1. 先把交互结构定住
2. 再接 provider 路由
3. 最后做视觉微调

### 2. `App.tsx` 仍然过大

目前主流程、状态、生成逻辑、白板交互、右侧联动都还集中在 `App.tsx`。  
这会带来几个问题：

- 改一个 UI 很容易碰坏另一个功能
- 很难做独立测试
- provider 路由越来越复杂后，继续堆在这里会失控

### 3. `@` 引用现在是可用版，不是最终版

目前恢复的 `@` 是基于 `textarea + 弹出列表` 的稳定实现，优点是输入法稳定，缺点也很明确：

- 引用 token 还是文本形式
- 用户手动改 token 文本后，映射关系会失效
- 还不是“真正结构化 token”

也就是说，它适合当前阶段先保证可用，但后面仍然要升级。

### 4. 多 provider 路由还不够完整

现在已经开始支持：

- LLM 润色按文本模型/provider 路由
- 图片生成按图片模型/provider 路由

但还不够完整，主要问题包括：

- 视频仍基本锁在 Google Veo
- 白板元素参考生成仍主要依赖 Google 路线
- 不同 provider 的错误提示和参数差异没有统一抽象
- 设置里虽然能配置能力，但前后端能力映射还比较粗

### 5. 代码和文案仍有历史债

当前还存在：

- 部分中文文案乱码
- 一些历史组件还在仓库里但已不是主路径
- README 和真实代码状态容易再次漂移

这些问题虽然不一定立刻阻塞功能，但会持续影响协作和维护体验。

## 四、当前最需要优化的功能

这里按优先级分三层。

### P0：必须优先做的

#### 1. 把 `首尾帧` 做成真正独立流程

现在的 `首尾帧` 更像一个入口标记，还不是完整工作流。  
下一步应该补齐：

- 首帧 / 尾帧上传或选择
- 首尾帧预览
- 专用参数区
- 真正调用对应的视频生成链路

#### 2. 完善多 provider 能力抽象

目标不是“支持更多名字”，而是让这几类能力稳定：

- 文本润色
- 文生图
- 参考图生图
- 视频生成

要做的包括：

- provider 能力表
- 统一错误处理
- 不同 provider 参数映射
- 设置页的能力校验

#### 3. 把 `@` 引用升级成结构化数据

现在它能用，但还不够稳。  
后面应该升级成真正的结构化 token，而不是依赖纯文本占位。

### P1：接下来一轮要做的

#### 4. 重构 `App.tsx`

建议拆成至少四层：

- `canvas state`
- `generation state`
- `panel/ui state`
- `provider settings`

这一步做完，后面继续迭代会轻松很多。

#### 5. 清理底部主输入框的展开交互

现在的方向已经对了，但还不够精炼。  
可以继续优化成：

- 默认只显示 3 到 4 个核心按钮
- 展开后展示细项
- 不同模式下自动切换只相关的选项

#### 6. 清理文案和编码问题

这已经不是“美化问题”，而是会直接影响产品观感和团队效率的问题。  
建议尽快统一所有中文文案编码和输出规范。

### P2：中期优化

#### 7. 本地存储从 `localStorage` 迁移

当前 API、素材、生成历史大多存在 `localStorage`，短期够用，但中期一定不够。  
下一步应考虑：

- IndexedDB
- 更稳定的项目保存结构
- 历史记录清理策略

#### 8. 项目保存 / 导出 / 作品集

当前“生成历史”已经有了，但“项目级存档”和“作品级输出”还不完整。  
这会直接限制产品从原型走向真正可用。

#### 9. 拆包和性能优化

Vite 构建已经持续提示包体较大。  
后面需要：

- 路由或面板级拆包
- 延迟加载非核心功能
- 减少一次性加载的编辑器与 AI 模块

## 五、建议的下一阶段开发顺序

建议不要再同时推太多方向，按下面顺序更稳：

1. `首尾帧` 完整工作流
2. 多 provider 路由补齐与错误处理统一
3. `@` 结构化引用升级
4. `App.tsx` 拆分
5. 本地存储与项目保存升级
6. 文案编码与 UI 细节清理

## 六、结论

当前版本最值得肯定的地方，是产品的主交互骨架已经开始成型。  
当前版本最需要警惕的地方，是功能越来越多，但抽象层和状态边界还不够稳。

换句话说，下一阶段最重要的不是“再多做几个按钮”，而是把：

- 主输入工作流
- provider 路由
- 数据持久化
- 状态结构

这四件事真正打牢。
